# AO Process — Deployment & Fallback Guide

This guide covers deploying, bootstrapping, and operating the PermaBrain AO process, plus fallback strategies when AO is unavailable.

## Overview

PermaBrain uses an [AO](https://ao.arweave.dev/) process as a real-time index layer on top of Arweave permanent storage. The architecture is:

```
┌─────────────────────────────────────────────────┐
│                  PermaBrain                     │
├─────────────────┬───────────────────────────────┤
│  CompositeTransport                            │
│  ┌───────────┐  ┌──────────┐  ┌──────────────┐ │
│  │ AO (dryrun)│→ │ Arweave  │→ │ Local Cache  │ │
│  │ instant    │  │ GraphQL  │  │ fallback      │ │
│  │ free reads │  │ perm     │  │ offline       │ │
│  └───────────┘  └──────────┘  └──────────────┘ │
├─────────────────┴───────────────────────────────┤
│  Writes: Arweave (permanent) + AO (index)      │
└─────────────────────────────────────────────────┘
```

**Key principle:** Arweave is the source of truth. AO is a fast index that can be rebuilt from Arweave at any time.

## Prerequisites

- Node.js 24+
- An Arweave JWK wallet (generated by `permabrain init`)
- AO process ID (from `permabrain ao-deploy` or a shared process)

## Deployment

### 1. Initialize PermaBrain

```bash
permabrain init --json
```

This creates:
- `~/.permabrain/config.json` — transport and gateway configuration
- `~/.permabrain/keys.json` — your Arweave JWK identity
- `~/.permabrain/cache/` — local article and attestation cache

### 2. Spawn an AO Process

```bash
permabrain ao-deploy --json
```

This:
1. Spawns a new AO process using the AOS module
2. Saves the process ID to `config.json`
3. Loads `process.lua` into the process
4. Returns the process ID, module ID, and scheduler ID

**Custom module/scheduler:**

```bash
permabrain ao-deploy --module <module-id> --scheduler <scheduler-id> --json
```

### 3. Bootstrap the Process

```bash
permabrain ao-bootstrap --json
```

This:
1. Waits for the AO process to be ready (polls with `Info` dryrun)
2. Loads `process.lua` (or re-loads if updating)
3. Syncs existing articles and attestations from Arweave

**Using an existing process:**

```bash
permabrain ao-bootstrap --process <existing-process-id> --json
```

### 4. Configure Environment

Set environment variables or update `config.json`:

```bash
# Environment variables
export PERMABRAIN_TRANSPORT=ao
export PERMABRAIN_AO_PROCESS_ID=<your-process-id>

# Optional: custom AO endpoints
export PERMABRAIN_AO_MU_URL=https://mu.example.com
export PERMABRAIN_AO_CU_URL=https://cu.example.com
export PERMABRAIN_AO_GATEWAY_URL=https://arweave.net
export PERMABRAIN_AO_GRAPHQL_URL=https://arweave.net/graphql
```

Or in `config.json`:

```json
{
  "transport": "ao",
  "ao": {
    "processId": "your-process-id",
    "muUrl": "https://mu.example.com",
    "cuUrl": "https://cu.example.com"
  }
}
```

## Usage

### Query Articles (AO dryrun — instant, free)

```bash
permabrain ao-query --topic computing --json
permabrain ao-query --kind person --json
permabrain ao-query --key person/ada-lovelace --json
```

### Get Article Metadata (AO dryrun)

```bash
permabrain ao-get person/ada-lovelace --json
```

### Get Consensus Score (AO dryrun)

```bash
permabrain ao-consensus person/ada-lovelace --json
```

### Sync Arweave Data to AO

```bash
permabrain ao-sync --json
```

This fetches all articles and attestations from Arweave and sends a `Sync` message to the AO process. Use this when:
- Bootstrapping a new process
- Re-indexing after AO process issues
- Adding data that was published while AO was offline

## Transport Modes

### AO Transport (`transport: "ao"`)

Reads via AO dryrun (instant, free). Falls back to Arweave GraphQL for `queryByTags` and `fetchDataItem`. Writes go to Arweave first, then AO.

```json
{ "transport": "ao", "ao": { "processId": "..." } }
```

### Composite Transport (`transport: "composite"`)

Cascading reads: AO → Arweave → local cache. Writes go to Arweave + AO in parallel.

```json
{ "transport": "composite", "ao": { "processId": "..." } }
```

### Arweave Transport (`transport: "arweave"`)

Direct Arweave GraphQL. No AO involvement.

```json
{ "transport": "arweave" }
```

### Local Transport (`transport: "local"`)

File-based cache only. No network calls.

```json
{ "transport": "local" }
```

## Fallback Behavior

### Read Operations

| Method | Primary | Fallback 1 | Fallback 2 |
|--------|---------|------------|------------|
| `queryArticles` | AO dryrun | Arweave GraphQL | — |
| `getArticle` | AO dryrun | Arweave GraphQL | — |
| `getConsensus` | AO dryrun | — (returns null) | — |
| `queryByTags` | Arweave GraphQL | Local cache | — |
| `fetchDataItem` | Arweave gateway | Local cache | — |

### Write Operations

| Method | Primary | Secondary |
|--------|---------|-----------|
| `uploadDataItem` | Arweave (permanent) | AO message (index) |
| `sendMessage` | AO message | — |

### Error Handling

- **AO dryrun fails:** `queryArticles` falls back to Arweave GraphQL. `getArticle` and `getConsensus` return `null`.
- **Arweave fails:** Falls through to local cache (composite transport) or throws.
- **AO message fails (write):** The Arweave upload still succeeds. AO failure is logged as a non-fatal error.

### Manual Re-sync

If AO process state diverges from Arweave:

```bash
# Full re-sync from Arweave
permabrain ao-sync --json

# Re-bootstrap (re-loads Lua + syncs)
permabrain ao-bootstrap --json
```

## process.lua Handlers

The AO process (`process.lua`) implements these handlers:

| Handler | Action | Input | Output |
|---------|--------|-------|--------|
| `Publish` | Create/update article | Article metadata tags | `Publish-Notice` |
| `Attest` | Record attestation | Attestation metadata tags | `Attest-Notice` |
| `Query` | List articles by filters | Topic/Kind/Key tags | `Query-Response` (JSON array) |
| `Get` | Get single article | `Article-Key` tag | `Get-Response` (JSON object) |
| `Consensus` | Get consensus score | `Article-Key` tag | `Consensus-Response` (JSON) |
| `Sync` | Bulk load data | JSON payload (articles + attestations) | `Sync-Notice` |
| `Info` | Process info | — | `Info-Response` (JSON) |

### Version Bumping

When an article with the same key but higher version is published, the process automatically:
- Increments the version number
- Links `previousId` to the prior article
- Maintains `rootId` pointing to the original

### Consensus Scoring

Consensus follows the same algorithm as the JS codebase:

| Opinion | Weight |
|---------|--------|
| valid | +1.0 |
| partially-valid | +0.5 |
| invalid | -1.0 |
| disputed | -0.75 |
| outdated | -0.5 |

- Only the latest attestation per agent is counted (deduplication)
- Attestations targeting non-latest versions get 0.5× weight
- Score = weighted sum ÷ total weight

## Identity Requirements

- **AO writes (message/dryrun):** Require an Arweave JWK identity (`arweave-rsa4096`). Ed25519 is not yet supported for AO.
- **Arweave uploads:** Support both `arweave-rsa4096` and `ed25519` key types.

To create an Arweave identity:

```bash
permabrain init --key-type arweave-rsa4096 --json
```

## Troubleshooting

### AO process not responding

```bash
# Check process status
permabrain ao-bootstrap --process <process-id> --json
```

Wait 30-60 seconds after spawning — AO processes need time to initialize on-chain.

### Stale data in AO

```bash
# Re-sync from Arweave
permabrain ao-sync --json
```

### AO transport falling back to Arweave

Check logs for `AO dryrun failed` messages. Common causes:
- Process not yet initialized
- Network issues with CU/MU endpoints
- Process was evicted (re-deploy needed)

### Switching transports

```bash
# Use Arweave directly (no AO)
export PERMABRAIN_TRANSPORT=arweave

# Use composite (best resilience)
export PERMABRAIN_TRANSPORT=composite
export PERMABRAIN_AO_PROCESS_ID=<process-id>
```

## Architecture Notes

- AO process state is **ephemeral** — it can be rebuilt from Arweave at any time
- Arweave is the **permanent** layer — all data lives there first
- The `Sync` handler enables bulk re-indexing from Arweave
- Dual-write (Arweave + AO) ensures data survives AO process failures
- Consensus scores in AO are computed on-demand via dryrun (no gas cost)
