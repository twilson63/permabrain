PermaBrain README Review — Product Requirements Document

Task Summary

Conduct a full open-source readiness review of the PermaBrain project (v0.2.0) and bring the README (and missing open-source artifacts) up to modern open-source standards. The project is a local-first CLI + HTTP API + web viewer for a public signed knowledge graph on Arweave/HyperBEAM, with 82 source modules, 157 tests, Pi skills, and a PWA viewer.

Definition

"Open-source readme standards" in this project means: a README that enables any developer to understand what the project is, install it, use it, contribute to it, and trust its safety — without needing prior knowledge. The README should be paired with standard open-source artifacts (LICENSE, CONTRIBUTING.md, CODE_OF_CONDUCT.md, SECURITY.md) that are referenced from the README itself.

What it does NOT mean: rewriting source code, changing the CLI API, adding features, or altering the viewer. This is documentation + artifact generation only.

Goals

  1. Identify all gaps between the current README and modern open-source standards
  2. Create missing open-source artifacts (LICENSE, CONTRIBUTING.md, CODE_OF_CONDUCT.md, SECURITY.md)
  3. Restructure and improve the README for clarity, completeness, and discoverability
  4. Ensure the README references all new artifacts
  5. Fix any inconsistencies between the README, CHANGELOG, and actual project state

Non-Goals

  • Do not modify any source code in src/, scripts/, or viewer/
  • Do not change the CLI surface or package.json structure
  • Do not add new features or change the architecture
  • Do not rewrite the entire README — improve incrementally with clear acceptance criteria per step
  • Do not remove existing content unless it is actively wrong or misleading

Constraints

  • All new artifacts must be created in the project root (matching convention)
  • LICENSE must be compatible with npm packaging (currently missing)
  • README must remain a single file; do not fragment it across sub-pages
  • All references to features must match actual implementation (verify against src/ files)
  • No secrets, private keys, or credentials in any artifact
  • Keep the existing tone: direct, practical, no marketing fluff

Risks

Context.md is outdated. The context.md file says "no implementation files exist" — the project has 82 source modules. This file should be noted as stale but not part of this task's scope.
CHANGELOG has duplicate section headers. Two "### Added" sections under [Unreleased].
No LICENSE exists. Without one, the project is not actually open source. Must pick an appropriate license.
